# UpdateManager

**UpdateManager** — Windows-утилита для создания и доставки патчей приложения с помощью **SimplePatchTool** и встроенного **SelfPatcher**.

Приложение помогает подготовить проект обновлений, добавить новые версии билда, собрать патч и доставить готовые файлы в локальную папку или на FTP-сервер.

## Возможности

* Создание проекта обновлений.
* Открытие существующего проекта.
* Автоматическое определение версии билда по `FileVersion` главного `.exe`.
* Ручной выбор главного исполняемого файла.
* Сборка repair / installer / incremental патчей.
* Поддержка self-patching приложения через встроенный `SelfPatcher`.
* Доставка готового патча:

  * в локальную папку;
  * на FTP-сервер.
* Проверка опубликованного патча через `VersionInfo.info`.
* Хранение списка недавних проектов.
* Поддержка светлой и тёмной темы.

## Требования

* Windows.
* .NET Framework 4.8.
* Visual Studio с поддержкой проектов .NET Framework.
* Для сборки проекта должны быть доступны библиотеки из папки `lib`:

  * `SimplePatchTool.dll`
  * `SimplePatchToolSecurity.dll`
  * `FluentFTP.dll`

## Сборка проекта

1. Склонируйте репозиторий:

```bash
git clone https://github.com/BorisBaNaNa/UpdateManager.git
```

2. Откройте решение:

```text
UpdateManager.sln
```

3. Убедитесь, что папка `lib` находится рядом с папкой `UpdateManager`.

4. Соберите проект в конфигурации `Debug` или `Release`.

После сборки исполняемый файл будет находиться в:

```text
UpdateManager/bin/Debug/
```

или

```text
UpdateManager/bin/Release/
```

## Основной сценарий использования

### 1. Создание проекта обновлений

1. Запустите `UpdateManager.exe`.
2. Нажмите **Создать проект**.
3. Выберите папку, в которой будет храниться проект обновлений.
4. Введите имя проекта.

После создания в выбранной папке появятся служебные файлы и директории проекта:

```text
Settings.xml
Versions/
Output/
SelfPatcher/
Other/
updatemanager.project.xml
```

> Не рекомендуется создавать проект обновлений внутри папки с билдом приложения. Папка проекта должна быть отдельной.

## 2. Настройка проекта

Откройте настройки проекта и заполните основные параметры.

### BaseDownloadURL

`BaseDownloadURL` — базовый URL, по которому клиентское приложение будет скачивать файлы патча.

Пример:

```text
https://example.com/patches/MyApp/
```

Важно: этот URL встраивается в патч во время сборки. Если изменить `BaseDownloadURL` после сборки, патч нужно собрать заново.

### Типы патчей

Можно включить один или несколько типов патчей:

* **Repair Patch** — полный патч для восстановления файлов.
* **Installer Patch** — патч для установки.
* **Incremental Patch** — инкрементальный патч между версиями.

Для обычного обновления приложения чаще всего нужен **Incremental Patch**. Для большей надёжности можно дополнительно включить **Repair Patch**.

### Self Patching App

Включите `IsSelfPatchingApp`, если приложение должно обновлять само себя.

При создании проекта UpdateManager автоматически копирует встроенный `SelfPatcher` в папку проекта `SelfPatcher/`.

## 3. Выбор папки билда

1. Нажмите **Выбрать билд**.
2. Укажите папку с готовым билдом приложения.
3. UpdateManager попробует найти главный `.exe` и прочитать его `FileVersion`.

По умолчанию главный исполняемый файл ищется по имени проекта:

```text
<ИмяПроекта>.exe
```

Например:

```text
MyApp.exe
```

Если главный `.exe` называется иначе, выберите его вручную через кнопку выбора исполняемого файла.

## 4. Версия билда

Версия берётся из `FileVersion` главного `.exe`.

Допустимый формат версии:

```text
1.0.0.0
```

Версия должна состоять только из чисел, разделённых точками.

Недопустимые примеры:

```text
v1.0
1.2-beta
1.0.0f1
```

Новая версия не должна быть ниже уже собранной максимальной версии, потому что инкрементальные патчи строятся по возрастанию версий.

## 5. Создание патча

1. Выберите папку билда.
2. Проверьте или введите версию.
3. Нажмите **Создать патч**.
4. Дождитесь завершения операции.

UpdateManager скопирует билд в:

```text
Versions/<version>/
```

Затем SimplePatchTool соберёт патч в папку:

```text
Output/
```

Например:

```text
Versions/1.0.0.0/BuildFiles...
Output/PatchFiles...
```

## 6. Доставка патча

После успешной сборки патч нужно доставить туда, откуда клиентское приложение сможет его скачать.

Нажмите **Доставить патч** и выберите способ доставки.

### Доставка в локальную папку

Выберите метод:

```text
Папка (локальный путь)
```

Затем укажите папку назначения.

UpdateManager скопирует содержимое `Output/` в выбранную папку.

Этот вариант удобно использовать, если вы вручную загружаете файлы на сервер или тестируете патчи локально.

### Доставка на FTP

Выберите метод:

```text
FTP (сервер)
```

Укажите параметры подключения:

* хост;
* порт;
* логин;
* пароль;
* удалённую папку.

UpdateManager загрузит содержимое `Output/` на FTP-сервер.

Пароль FTP сохраняется в защищённом виде через механизм Windows DPAPI. Если открыть проект на другом компьютере или под другой учётной записью Windows, пароль может потребоваться ввести заново.

## 7. Проверка патча

После публикации можно выполнить проверку.

Проверка использует URL:

```text
<BaseDownloadURL>/VersionInfo.info
```

Например:

```text
https://example.com/patches/MyApp/VersionInfo.info
```

UpdateManager попробует скачать патч во временную папку и проверить, что опубликованные файлы доступны.

## Рекомендуемый порядок работы

1. Собрать новую версию приложения.
2. Убедиться, что у главного `.exe` выставлен корректный `FileVersion`.
3. Открыть проект в UpdateManager.
4. Выбрать папку нового билда.
5. Проверить определённую версию.
6. Создать патч.
7. Доставить патч в папку или на FTP.
8. Выполнить проверку опубликованного патча.

## Важные ограничения

* Папка билда не должна совпадать с папкой проекта обновлений или находиться внутри неё.
* `BaseDownloadURL` должен быть задан до сборки патча.
* После изменения настроек проекта патч нужно пересобрать.
* Версии должны идти по возрастанию.
* Формат версии должен быть числовым.
* Для self-patching приложения в проекте должен присутствовать `SelfPatcher`.

## Структура проекта обновлений

Пример структуры после создания проекта:

```text
MyUpdateProject/
├── Settings.xml
├── updatemanager.project.xml
├── Versions/
│   ├── 1.0.0.0/
│   └── 1.0.1.0/
├── Output/
├── SelfPatcher/
│   ├── SelfPatcher.exe
│   └── SelfPatcherCore.dll
└── Other/
```

### Settings.xml

Основной файл настроек SimplePatchTool.

### updatemanager.project.xml

Дополнительный файл UpdateManager. Используется для хранения служебной информации, например последней выбранной папки билда и настроек доставки.

### Versions/

Папка с сохранёнными версиями приложения.

Каждая версия хранится в отдельной подпапке:

```text
Versions/1.0.0.0/
Versions/1.0.1.0/
```

### Output/

Папка с готовыми файлами патча после сборки.

Именно содержимое этой папки нужно публиковать на сервере.

### SelfPatcher/

Папка со встроенным self-patcher'ом, который нужен для обновления self-patching приложения.

## Частые ошибки

### Не задан BaseDownloadURL

Перед сборкой патча нужно указать `BaseDownloadURL` в настройках проекта.

Без него клиентское приложение не сможет скачать файлы обновления.

### UpdateManager не нашёл версию

Проверьте, что:

* выбран правильный билд;
* главный `.exe` находится в папке билда;
* имя `.exe` совпадает с именем проекта или выбран вручную;
* у `.exe` заполнен `FileVersion`.

### Версия недопустима

Используйте только числовой формат:

```text
1.0.0.0
```

Не используйте буквы, префиксы и суффиксы.

### Версия ниже уже собранной

Нельзя собрать версию ниже максимальной версии из `Versions/`.

Например, если уже есть:

```text
1.0.2.0
```

то версия:

```text
1.0.1.0
```

будет отклонена.

### Output пустой

Сначала нужно создать патч. Только после этого его можно доставить.
